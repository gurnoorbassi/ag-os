# Data Classification

## Purpose

This document defines the data sensitivity levels AG Digitalz OS must use before handling records, memory, audit events, project records, connector output, runtime state, or future production systems.

## Levels

| Level | Meaning | Foundation Handling |
| --- | --- | --- |
| `public` | Approved public information, public docs, and non-sensitive published material. | Allowed. |
| `internal` | AG Digitalz operating metadata that is not secret, customer-specific, or production-derived. | Allowed when production-clean. |
| `confidential` | Sensitive business information that is not production data, customer data, or credentials. | Blocked unless owner approved for exact scope. |
| `restricted` | Information where exposure could harm AG Digitalz, customers, systems, finances, or operations. | Blocked unless owner approved for exact scope. |
| `customer_data` | Client, lead, user, or customer-specific information. | Blocked in foundation mode. |
| `production_data` | Data copied from, generated by, or controlling a production system. | Blocked in foundation mode. |
| `credentials_secrets` | Credentials, tokens, API keys, private keys, passwords, certificates, database URLs, signing secrets, and secret-store material. | Always blocked from the repo. |

## Handling Rules

- Schemas and docs may describe sensitive categories, but must not include real sensitive values.
- Customer data, production data, credentials, restricted information, or confidential information must not be used in examples, fixtures, tests, memory records, or docs during foundation mode unless the owner explicitly approves the exact handling path.
- Internal metadata must be minimal and source-controlled only when it helps governance.
- Secrets must live outside this repository in an approved secret manager or platform-specific secret store after a future approval.
- If classification is unclear, treat the data as the more sensitive level.
- Redacted data must be reviewed as a new artifact; redaction alone does not make data public or internal.

## Review Triggers

Security review is required when a change touches:

- Authentication or authorization.
- Webhooks or public endpoints.
- Database access.
- File uploads or exports.
- Customer data.
- Production data.
- Agent write permissions.
- Paid API actions.
- Credentials or secret handling.

## Data Downgrade Rule

Data classification may not be downgraded without owner approval and a documented reason. Redaction does not automatically make data safe; the redacted artifact must be reviewed on its own.

## Repo Handling Rule

The repository may contain `public` and production-clean `internal` data. All other classes require owner-approved scope and must never include credentials, customer records, production exports, or private contact details in the repository.

## Legacy Label Mapping

Some early schemas may still use older labels until a future schema migration. Map them as follows before activation:

- `none` maps to `public` when the content is truly non-sensitive.
- `secret` maps to `credentials_secrets`.
- `customer` maps to `customer_data`.
- `regulated` maps to `restricted` unless a stricter future regulated-data policy exists.

No live record may rely on an unmapped legacy label.
